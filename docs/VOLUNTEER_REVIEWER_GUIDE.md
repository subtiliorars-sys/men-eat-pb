# Volunteer reviewer guide

Thank you for helping review Men Eat Peanut Butter playtest feedback.

Reviewers do not need to decide the whole roadmap. Your job is to keep the feedback queue useful, kind, and actionable.

## Reviewer responsibilities

Review each submission and choose one:

- `approved` - actionable and useful for the backlog.
- `denied` - spam, abusive, duplicate with no new info, out of scope, or too vague.
- `needs_info` - potentially useful, but one specific detail is missing.

## What good feedback looks like

Approve feedback when it is:

- Specific enough to understand.
- About the current playable build.
- Useful for bugs, balance, game feel, clarity, accessibility, onboarding, or replayability.
- Safe to keep in a public/community queue.

Examples:

- "No Napkins feels too punishing because one miss seems to end the run spiral."
- "I did not notice Frenzy started until the HUD changed color."
- "The end screen should keep the modifier name visible."
- "Restart should be one click faster."

## When to mark `needs_info`

Use `needs_info` when the note might be valuable but needs one answer.

Good follow-up questions:

- "Which modifier were you using?"
- "Did this happen before or after Frenzy?"
- "Were you on mobile or desktop?"
- "What did you expect to happen instead?"

Avoid broad follow-ups like "Can you explain more?" Ask for one detail.

## When to deny

Deny feedback when it is:

- Spam or promotion.
- Abusive or unsafe for community review.
- A duplicate with no new detail.
- Unrelated to Men Eat Peanut Butter.
- A request that contradicts the design direction.
- Too vague after one needs-info attempt.

Examples:

- "This sucks." -> deny unless the player adds detail.
- "Add online multiplayer MMORPG mode." -> deny as out of scope.
- "Same as issue #12" with no new detail -> deny/duplicate.

## Review procedure

1. Open the pending queue.
2. Read the player note.
3. Read the attached run summary.
4. Check whether a similar note already exists.
5. Choose `approved`, `denied`, or `needs_info`.
6. Add a short reviewer note.
7. Move on.

Reviewer notes should be short:

- "Approved: clear balance report for No Napkins."
- "Denied: duplicate of Frenzy clarity theme."
- "Needs info: asked whether this happened on mobile."

## Escalate to owner

Set the item aside for owner review if it involves:

- harassment, threats, or personal data
- payment/pricing complaints
- legal/copyright concerns
- requests involving another creator or community conflict
- anything that would change the game identity

Suggested note:

> Escalated for owner review: [short reason].

## First reviewer session checklist

- [ ] Read this guide.
- [ ] Open the reviewer/admin link.
- [ ] Set reviewer name.
- [ ] Confirm token works.
- [ ] Review one test item as `approved`.
- [ ] Review one test item as `denied`.
- [ ] Review one test item as `needs_info`.
- [ ] Ask owner where to report confusing cases.

## Reviewer code of conduct

- Be kind.
- Assume good faith for unclear notes.
- Do not expose private contact details.
- Do not argue with players.
- Do not promise changes.
- Do not approve feedback just because you personally agree with it; approve it because it is useful and actionable.
